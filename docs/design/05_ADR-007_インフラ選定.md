| 項目         | 内容                                                                                   |

| 置き換え対象 | 01\_基本設計書 §6.3 のインフラ行「Vercel(または同等: CloudFront+ECS)+ Neon/RDS」       |

## 1. コンテキスト

当初設計はVercel(または同等としてCloudFront+ECS)を想定していた。今回「**無料で使えるならCloudFront+ECSへ変更、難しければCloudflareを検討**」という要求が出たため、2026年7月時点の料金体系を調査した。

### 1.1 調査結果: AWS(CloudFront+ECS)側

| 事実                                                                                                                                                                   | 影響                                                                                                                                             |
| ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| 2025年7月15日以降の新規AWSアカウントは、従来の12ヶ月無料枠が廃止され、**最大$200のクレジット制(サインアップで$100+タスク達成で$100)・最長6ヶ月**の無料プランに移行した | 「無料期間」は恒久ではなく最長6ヶ月で終了。無料プランはクレジット枯渇か6ヶ月経過で終わり、アップグレードしなければアカウントは自動クローズされる |
| CloudFront自体は常時無料枠(**月1TB転送+1,000万リクエスト**)が存続                                                                                                      | 配信層(CDN)単体は無料で成立する                                                                                                                  |
| **ECSのコンピュート(Fargate/EC2)とALBには常時無料枠がない**。旧12ヶ月無料のEC2 750時間/月も2025年7月15日以降の新規アカウントには適用されない                           | 常時稼働のコンテナ+ロードバランサで、参考構成の実測ではEC2型で月$15前後、Fargate型で月$32前後の恒常コストが発生する                              |
| RDS等のDBも同様にクレジット消化対象                                                                                                                                    | DB分の恒常コストも加算                                                                                                                           |

**結論(AWS)**: CloudFront(配信)は無料でも、**ECS(実行層)+ALB+DBが恒常有料**であり、「CloudFront+ECSの無料運用」は6ヶ月のクレジット期間限定でしか成立しない。学習サービスとして継続運用する本件の要件に合致しない。

### 1.2 調査結果: Cloudflare側

| 事実                                                                                                                                                               | 影響                                                                                                                                                                                |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Workers Freeプラン: **10万リクエスト/日、CPU時間10ms/呼び出し、Workerサイズ3MiB(圧縮後)、サブリクエスト50/呼び出し**。**静的アセットへのリクエストは無料・無制限** | 本アプリはSSG中心(教材・可視化は全て静的アセット+クライアント実行)のため、Workerを消費するのは進捗系APIのみ。CPU時間はDB待ち(I/O)を含まないため、軽量JSON APIなら10msに収まる見込み |
| Next.jsは **OpenNextアダプタ(@opennextjs/cloudflare)** でWorkers上のNode.jsランタイムモードとして動作(メモリ128MB固定)                                             | フレームワーク変更は不要。ただしNode.js APIは互換レイヤ経由で一部制約あり                                                                                                           |
| Workers Paidは**$5/月**でサイズ10MiB・CPU上限拡大                                                                                                                  | 無料枠超過時の逃げ道が月額$5と極めて安価(Fargate構成の1/6以下)                                                                                                                      |
| 周辺: D1/KV/R2/Queues等が同プラットフォームに存在。R2には無料枠あり                                                                                                | S3相当はR2で代替可能                                                                                                                                                                |
| DB: Neon(PostgreSQL)無料ティアを併用                                                                                                                               | Postgres前提の既存設計(Prisma)を維持可能                                                                                                                                            |

**結論(Cloudflare)**: 本アプリの特性(**A-01により重い処理は全てユーザーのブラウザ内で実行され、サーバは軽量なJSON APIのみ**)とCloudflareの課金モデル(静的無料・Worker実行のみ課金)の相性が非常に良く、**恒常¥0運用が現実的**。

## 2. 検討した選択肢

| 選択肢                                                        | 恒常無料                                 | 評価                                                                                                                                                                                |
| ------------------------------------------------------------- | ---------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| A) CloudFront + ECS(Fargate/EC2)+ RDS                         | ✗(6ヶ月クレジットのみ。以後 月$30前後〜) | 要求条件「無料」を満たさない。運用負荷(コンテナ・ALB・パッチ)も本件には過剰                                                                                                         |
| B) CloudFront + Lambda(常時無料枠あり)+ DynamoDB              | △(構成次第で可)                          | サーバレス化でほぼ無料は可能だが、Next.js SSR/ISRのLambda+CloudFront自前構成は複雑度が高く、PostgreSQL前提の既存設計(Prisma/リレーショナル)をDynamoDBに書き換える改修コストが大きい |
| **C) Cloudflare Workers(OpenNext)+ 静的アセット + Neon + R2** | **◎(想定負荷では¥0、超過時も$5/月)**     | **採用**。SSG中心・クライアント実行という既存アーキテクチャ決定(A-01)と課金モデルが噛み合う。Next.js・Prisma・PostgreSQLの既存設計を最小改修で維持                                  |
| D) Vercel Hobby(無料)                                         | △                                        | 個人・非商用限定の規約制約があり、将来の公開運用と相性が悪い。比較基準として残置                                                                                                    |

---

## 3. 決定

**選択肢C: Cloudflareスタックを採用する。**

| レイヤ                 | 変更前(01 §6.3)               | 変更後                                                                                             |
| ---------------------- | ----------------------------- | -------------------------------------------------------------------------------------------------- |
| ホスティング/CDN       | Vercel(または CloudFront+ECS) | **Cloudflare Workers + 静的アセット配信**(OpenNextアダプタ、SSG成果物は静的アセットとして無料配信) |
| DB                     | Neon / RDS                    | **Neon 無料ティア(PostgreSQL)** + Prisma(driver adapter経由。必要に応じHyperdrive)                 |
| オブジェクトストレージ | S3                            | **Cloudflare R2**(修了証PDF等。無料枠内)                                                           |
| CI/CD                  | GitHub Actions(変更なし)      | 同左 + `wrangler deploy`                                                                           |

## 4. 帰結(Consequences)と設計書への反映事項

### 4.1 受け入れる制約とその対策

| #   | 制約                                                                                                                     | 対策 / 設計変更                                                                                                                                                                                                                   |
| --- | ------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| C-1 | Worker本体サイズ **3MiB(Free)**                                                                                          | サーババンドルの厳格管理。Monaco/sql.js/可視化は元々クライアント側静的アセットのため影響なし。**CIにサーババンドルサイズゲート(2.5MiB警告/3MiB失敗)を追加**(→ WBS T-002/T-402に反映)。超過が恒常化した場合はPaid($5/月)へ移行判断 |
| C-2 | CPU 10ms/呼び出し(Free)                                                                                                  | 進捗系APIは軽量JSONに徹する(既存設計どおり)。重い集計(ダッシュボード)はSQL側に寄せ、Worker内での大きなループを避ける                                                                                                              |
| C-3 | **Argon2ネイティブ実装が動かない**(workerdはネイティブアドオン不可)                                                      | 02 §2.1/§10のパスワードハッシュを「**Argon2id(純WASM実装)、導入不可時はscrypt(WebCrypto系)へフォールバック**」に改訂。OAuth(GitHub/Google)を推奨導線として維持                                                                    |
| C-4 | 修了証のサーバ側再検証(02 §7.5)が想定していた **Node worker_threads はWorkers上で実行不可**(CPU制限・分離要件的にも不適) | 再検証ジョブを **GitHub Actions の workflow_dispatch ジョブ**として実行する方式に変更(証明書API→Actionsをキック→結果をDBに書き戻し)。非同期化するがUX上は「発行申請→数分後に通知」で許容(01 §2で修了証はShould相当)               |
| C-5 | 10万リクエスト/日(Free)                                                                                                  | 静的アセットはカウント外のため、実質制約はAPI呼び出しのみ。進捗APIの呼び出し頻度設計(スクロール80%は1レッスン1回等)は既存どおりで十分収まる想定。閾値監視を運用項目に追加                                                         |
| C-6 | Node.js API互換レイヤの部分未対応                                                                                        | T-000 Walking Skeleton の検証項目に「**Cloudflare(wrangler preview)上でのAuth/Prisma/API動作確認**」を追加し、非互換を最初期に検出する                                                                                            |

### 4.2 変わらないこと

- Next.js / TypeScript / Prisma / PostgreSQL / MDXパイプラインという開発体験は維持(A-04は不変)
- 演習実行エンジンは元々100%クライアントサイド(A-01)のため**一切影響なし**
- アプリはインフラ非依存に保つ(R2アクセスは抽象化レイヤ経由)ため、将来AWSへ移行する場合もアプリ改修は限定的

### 4.3 撤退基準(この条件でPaid移行 or 再検討)

- サーババンドルが恒常的に3MiB超 → Workers Paid($5/月)
- API呼び出しが10万/日に接近(DAU換算で概ね数千規模) → Workers Paid
- Neon無料ティアの容量/コンピュート制限に到達 → Neon有料 or 再ADR

- 01\_基本設計書: §6.2にA-07行を追加、§6.3インフラ行を更新 ✅(反映済)
