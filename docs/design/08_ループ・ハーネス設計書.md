## 1. リサーチサマリ: 2つの思想の要点

### 1.1 Loop Engineering(ループエンジニアリング)

「エージェントに1ターンずつプロンプトを打つのをやめ、**プロンプトを生成・実行・検証しゴール到達まで繰り返すシステム(ループ)そのものを設計する**」思想。Claude Codeチームの定義では「**ループ=停止条件が満たされるまで作業サイクルを繰り返すエージェント**」。

| 要素                 | 内容                                                                                                                                                          |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ループの最小要件     | ①トリガー(何が起動するか) ②**検証可能なゴール**(機械が真偽判定できる終了状態)                                                                                 |
| 4つのループ型        | Turn-based(人間が毎ターン駆動)/ **Goal-based(`/goal`)** / Time-based(`/loop` `/schedule`)/ Proactive(イベント駆動の組み合わせ)                                |
| `/goal` の仕組み     | 完了条件を1度書くと、毎ターン後に**別の小型モデル(既定Haiku)が条件達成を判定**し、未達なら自動で次ターンへ。v2.1.139+                                         |
| `/goal` 評価器の制約 | **評価器はトランスクリプトしか見ない**(コマンド実行・ファイル読取をしない)。条件は「Claude自身の出力で証明できる形」で書く必須                                |
| 良い条件の3要素      | ①測定可能な1つの終了状態(テスト結果/exit code/ファイル数) ②証明方法の明示(`npm test` が exit 0) ③守るべき制約(触ってはいけないもの)                           |
| 構成部品(Osmani)     | Automations / Worktrees / Skills / Plugins(MCP) / **Sub-agents(実装者と検証者の分離)** + 外部メモリ(ディスク上の状態)                                         |
| 3つの本番問題        | ①無限ループ → `or stop after N turns` を必ず入れる ②ゴールドリフト → 制約明記+検証者分離 ③トークン爆発 → `/goal`(状態表示)・`/usage` で監視、評価は小型モデル |

### 1.2 Harness Engineering(ハーネスエンジニアリング)

LLMは**ステートレス**であり毎回「初めまして」なので、ルール・記憶・検証手段を**モデルの外側の実行環境として恒久的に整備する**思想。核心はHashimotoの「**エージェントがミスするたび、二度と同じミスをしないよう環境側に恒久修正を施す**」。Fowlerは**Guide(事前誘導=feedforward)とSensor(事後検知=feedback)**の2系統に体系化。

Anthropic自身のハーネス設計知見(公式2記事):
| 知見 | 内容 |
|---|---|
| 分解と外部状態 | initializerがfeature list/進捗ファイルを作り、コーディングエージェントが**1機能ずつ**進めてクリーンな状態でコミット・引き継ぎ資産を残す |
| **生成者/評価者の分離**(GAN型) | 自己評価は自分に甘くなる。**別コンテキストの懐疑的な評価者**を立て、基準に対して採点→生成者が具体的フィードバックに対して反復する方が調整しやすい |
| スプリント契約 | コードを書く前に生成者と評価者が「このチャンクのdoneとは何か」を**合意してから**実装する |
| 実操作による検証 | 評価者はPlaywright等で**実際にアプリを操作して**採点する(静的レビューでは表面的なバグしか捕まらない) |
| 主観品質の可格付け化 | 「良いデザインか」は判定不能でも「**この原則に従っているか**」は採点できる。基準+ハード閾値(1つでも下回れば不合格)を定義 |
| 部品の賞味期限 | **ハーネスの各部品は「モデルにはこれができない」という仮定の符号化**であり、その仮定はモデル改善で期限切れになる。新モデルが出たら部品を1つずつ外して負荷担持を再確認する |
| 制御の層(Steering) | 事実はCLAUDE.md、スコープ付き制約はRules、手順はSkills、隔離作業はSubagents、**確実に起こすべきことはHooks/Settings(deny)**。「テキストはお願い、コードは保証」 |
| LLM/スクリプトの責務分離 | 決定的処理(パース、I/O、集計)はスクリプトに。LLMは判断と生成に限定(トークン・安定性が大幅改善) |

## 3. 導入設計

### 3.1 【G1】タスク実行のGoal-based Loop化(`/goal` の標準装備)

**原則**: 受入基準が機械検証可能なタスク(=ほぼ全タスク)は、投入時に受入基準を `/goal` 条件へ変換し、**達成まで自走**させる。人間の関与は「投入」と「達成後のレビュー」の2点に縮退する。

**条件テンプレート(公式の3要素+上限に準拠)**:

```
/goal T-xxx が完了している。証明: (1) npm run lint && npm run typecheck &&
npm run test が exit 0 で、その出力がこの会話に表示されている
(2) <タスク固有の受入基準: 例「T-104のAPI統合テスト仕様表の全ケースが
テスト名として存在し合格している」> (3) ブランチ feat/T-xxx-… にコミットし
git push が成功している。制約: docs/design/03 の T-xxx 成果物以外のファイルを
変更しない。lib/contracts/ を変更しない。テストの skip/only/expect緩和をしない。
or stop after 25 turns
```

**タスク種別ごとの適用**(04 §5 の改訂):
| 種別 | ループ設計 |
|---|---|
| S/Mタスク(承認ゲートなし) | 投入プロンプト冒頭に `/goal` 条件を記載して `--bg` 投入 → 完全自走 |
| M/Lタスク(計画承認あり) | 通常投入→計画承認(人間 or 3.3の契約レビュー)→承認返信に「/goal <条件> を設定して続行」と書く2段階 |
| 修正ループ(旧T3) | **人間の3周ルールを廃止し `/goal` 化**: 「指摘テスト群が合格し既存テスト全緑、指摘外ファイル変更なし、or stop after 10 turns」。10ターンで未達なら初めて人間にエスカレーション(タスク再分割判断) |
| CI/PR監視 | マージ前PRのCI失敗対応は Time-based: `/loop 10m PR #N のCIを確認し、失敗していれば修正してpush。緑なら報告して停止` |

**暴走ガード(3つの本番問題への対策)**:

1. 全条件に `or stop after N turns` を必須化(S: 15 / M: 25 / L: 40 を既定)
2. 条件の「制約」節にスコープ・contracts・テスト誠実性を明記(ゴールドリフト対策)
3. `/goal`(引数なし)と `/usage` でトークン消費を確認する運用を07のC-4cに統合。評価は小型モデルなので評価コスト自体は軽微

**評価器制約への対応**: 評価器はトランスクリプトしか見ないため、CLAUDE.mdに「検証コマンドの実行結果は必ず出力に表示する(サイレント実行禁止)」を追加する(→ §4のCLAUDE.md v2 規則11)。

### 3.2 【G2】敵対的QA評価者(qa-evaluator)の導入 — 生成者/評価者分離の完成

公式ハーネス記事の核心(「自己評価は甘い」「実操作で検証」「懐疑的に調整」)を、サブエージェントとして実装する。

- 定義: `.claude/agents/qa-evaluator.md`(本書とセットで配布)
- 性格づけ: **「問題を見つけることが仕事。賞賛は成果ではない」**という懐疑的プロンプト。公式記事の教訓(「問題を見つけても大したことないと自己弁護して承認してしまう」)への対策として、**発見した問題を却下する場合は却下理由の明記を義務化**
- 検証方法: dev server起動→**実際にブラウザ操作**(Playwright MCP導入時)またはE2Eテスト+curl等での実挙動確認→エッジケースを意図的に突く(空入力、巨大入力、言語切替中の操作、リロード)
- 採点: §3.4の基準表に対し各観点1〜5点、**ハード閾値=全観点4以上**。1つでも未満なら差し戻し(修正は生成者側が行う。評価者は直さない)
- 適用タイミング: ①UI系タスク(T-103/106/108/112、Viz全部、S-02〜S-12)の完了報告前に必須 ②Phase統合検証(04 T5)の第一段として実行し、人間レビューは評価者レポートの確認に軽量化

**Playwright MCP(任意だが推奨)**: `claude mcp add playwright -- npx @playwright/mcp@latest` で導入し、qa-evaluatorが実ブラウザを操作できるようにする。未導入時はE2Eテスト実行+HTMLレスポンス検査にフォールバック(agent定義に両モード記載済み)。

### 3.3 スプリント契約(計画承認のエージェント化)

M/Lタスクの計画承認を「生成者が計画を書く→**qa-evaluatorが受入基準との整合を審査**→合意文書を`docs/tasks/contracts/T-xxx.md`に保存→人間は合意文書を一瞥して承認」に変更。公式記事のスプリント契約(実装前にdoneの定義を合意)を輸入しつつ、最終承認権は人間に残す。合意文書は外部メモリとして次セッション(修正時)の文脈にもなる。

### 3.4 【G3】主観品質のグレーディング基準

「良い教材か」を「この基準に従っているか」に変換する(公式のfrontend基準4観点の翻案)。qa-evaluatorと生成者の**双方のプロンプトに与える**(基準を知って作る方が初回品質も上がる、という公式の観察に従う)。

| 観点           | 5点の状態                                                        | 不合格例                                                |
| -------------- | ---------------------------------------------------------------- | ------------------------------------------------------- |
| 教材の明晰さ   | 前提知識を仮定せず、具体例→抽象の順で、1レッスン1論点            | 用語の未定義使用、原著を読んだ前提の説明                |
| 可視化の説明力 | 操作した結果が概念の理解に直結する(例: vnodes増→偏り減が見える)  | 動くだけで洞察が生まれない装飾アニメ                    |
| 操作性         | 初見でも迷わず主要操作に到達、キーボード完結、エラー時の回復導線 | 発見不能な機能、操作後の無反応                          |
| i18n品質       | ja/en が同一構造・同等の情報量で、切替後も状態保持               | 片言語の情報欠落、切替でエディタ内容消失                |
| 誠実性(コード) | テスト・実装が受入基準を実質的に満たす                           | モック偽装、テスト弱体化(test-integrity-reviewerと連携) |

### 3.5 【G4】強制層(Sensor)の追加 — 「テキストはお願い、コードは保証」

| 層              | 導入内容                                                                                                                                                        | ファイル                                    |
| --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------- |
| Settings(deny)  | 破壊的操作の静的ブロック: force push / ブランチ削除 / `rm -rf` の広域 / DBリセット / シークレット読出しの露骨なパターン                                         | `.claude/settings.json`(本書とセットで配布) |
| Settings(allow) | 最初は最小にし、**「確認が出る→許可パターンを追記」のサイクルで実運用で磨く**(先回りで広げない)                                                                 | 同上                                        |
| Rules           | CLAUDE.md肥大化の防止: i18n規約(`globs: content/**, messages/**`)とAPI規約(`globs: app/api/**`)を`.claude/rules/`へ分離し、該当ファイルを触るときだけ遅延ロード | `.claude/rules/i18n.md` ほか(T-001で作成)   |
| Hooks           | **当面導入しない**(denyで足りる範囲はdenyで。動的判断が必要になった時点で導入を検討) — 段階導入原則                                                             | —                                           |

### 3.6 【G6】検証スキルの整備

公式loops記事の「検証手順をSKILL.mdに符号化し、Claudeが自分の仕事をend-to-endで確認できるようにする」を導入。`.claude/skills/verify-webapp/SKILL.md`(本書とセットで配布)に本アプリ用の検証手順(dev server起動→対象ページ操作→console 0エラー→言語切替後の状態保持→受入コマンド)を定義。タスクプロンプトから検証手順の記述が消え、「verify-webappに従って検証」で済むようになる。

あわせて**LLM/スクリプト責務分離**を原則化: 決定的検証(slug整合、リンク切れ、バンドルサイズ)はスクリプト(`validate:content`等)が担い、LLMは判断が要る検証(基準採点、実操作)のみ行う。既存設計はこの原則に概ね合致しており、明文化のみ。

### 3.7 【G5/G7】ハーネス運用の2原則(CLAUDE.md v2へ)

1. **恒久修正原則(Hashimoto)**: エージェントの失敗を個別修正で終わらせない。同種の失敗を防ぐ恒久対策(CLAUDE.md/Rules/deny/スキル/スクリプトのどれに書くか)を検討し、STATUS.mdの決定事項ログに「失敗→恒久対策」を1行で記録する。**対策先の選び方**: 常時知るべき事実→CLAUDE.md / 特定領域の規約→Rules / 手順→Skills / 確実に止めるべき操作→deny
2. **部品見直し原則(Anthropic)**: ハーネスの各部品は「モデルへの不信の符号化」であり期限がある。**新しいモデルに切り替えた時**と**各Phase完了時**に、承認ゲート・ターン上限・評価者の適用範囲を1つずつ緩めて試し、不要になった足場を撤去する(07のD項に追加)

### 3.8 作業者用スキルの整備計画(Skills Roadmap)

**結論: verify-webapp 1本では不足。ただし全部を先回りで書かない。**

公式のスキル設計原則: スキルは進行的開示(Progressive Disclosure)で動くため、起動時に読み込まれるのは名前とdescriptionのみ(数トークン)で、本文はタスクが合致した時だけ、参照ファイルは実際に開かれた時だけロードされる。つまり**スキルを多数置いてもコンテキストペナルティはほぼない**。一方で、descriptionは「何をするか+いつ使うか」を具体語で書き、本文は500行未満、詳細はreferenceファイルへ、決定的処理はスクリプトへ、検証ループ(validator実行→修正→再実行)を手順に組み込む、が公式の品質基準。

本プロジェクトのタスクは**反復ファミリー**が明確(教材24本、Viz5本、API多数)なので、スキル化のROIが高い。ただし作成タイミングは**「最初の1本が人間レビューを通って完成した直後に、その完成品からパターンを抽出する」**を原則とする(証明されていないパターンの符号化はゴールドリフトの温床。恒久修正原則の応用)。

## 5. 結論(質問への回答)

**現設計は両思想を「骨格レベルでは」すでに含んでいた**。検証可能な受入基準(P2)、実装者/検証者分離の萌芽(サブエージェント)、外部メモリ(STATUS.md/設計書)、worktree隔離、スコープ制約は、Loop/Harness Engineeringの語彙で説明できる設計になっている。

一方で決定的な欠落は2点あった。①**ループの駆動者が人間のまま**だったこと(修正ループを人間が回す設計= Turn-basedループ止まり)。②**ハーネスが「お願い層」偏重**で、強制層(deny)と実操作する敵対的評価者を欠いていたこと。本書の導入により、タスクは「/goalで受入基準達成まで自走し、懐疑的な評価者に採点され、破壊的操作は環境が物理的に止め、失敗は環境側への恒久修正として蓄積される」構造になり、人間の役割は投入・契約承認・最終レビュー・エスカレーション判断に純化される。
