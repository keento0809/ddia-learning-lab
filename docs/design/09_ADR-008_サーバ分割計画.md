# ADR-008: サーバ分割(2 Worker構成)とサイズ予算制
## DDIA Learning Lab — Worker単位バンドル制約への構造的対応

| 項目 | 内容 |
|---|---|
| ステータス | Accepted (2026-07-19) |
| 背景となる事象 | T-112時点でサーババンドルがgzip 2.985MiBに達しFree枠3MiBに接触。Bundle Slim(Prisma compilerBuild="small")で2.252MiBへ削減したが、残余裕0.75MiBに対しSentryサーバSDK導入が未了 |
| 置き換え/補完対象 | ADR-007(インフラ選定)を補完。撤退基準§4.3の「Paid移行」より前に試す構造的対策として位置づけ |
| 関連 | 02 §3(API設計)、07 E-5(撤退基準) |

---

## 1. 反省: 元設計の欠陥の正確な特定

単一Worker構成自体はモノリス優先の定石に沿った妥当な出発点だった。真の欠陥は以下の2点であり、本ADRはその恒久対策を含む。

| 欠陥 | 恒久対策(本ADRに内包) |
|---|---|
| デプロイ単位ごとの**サイズ予算**をアーキテクチャの一級関心事として設計しなかった(ADR-007ではリスク1行扱い) | §4のサイズ予算表を正式なアーキテクチャ制約とし、CIでWorker別に強制する |
| T-000(Walking Skeleton)の検証項目に**実依存込みのデプロイサイズ実測**を含めなかった | 今後のインフラ系ADRでは「主要依存を入れた状態での資源実測」を受入基準に必須化(STATUS.md決定事項ログに記録済み) |

---

## 2. 決定: 2 Worker構成 + service binding

3分割(app/auth/api)ではなく**2分割**とする。Auth.jsはNext.jsのルーティングと密結合であり、認証を別Workerに切り出すとコストが利得を上回るため。

```
                    ┌─────────────────────────────────────┐
  ユーザー ────────▶│ worker-app (公開)                    │
                    │  Next.js (OpenNext) + 静的アセット     │
                    │  Auth.js (/api/auth/*, JWTセッション)  │
                    │  Prisma依存なし                       │
                    └───────────┬─────────────────────────┘
                                │ service binding(内部呼び出し・
                                │ リクエスト課金なし)
                    ┌───────────▼─────────────────────────┐
                    │ worker-api (非公開・binding経由のみ)   │
                    │  Hono + lib/contracts(zod)            │
                    │  Prisma + Neon                        │
                    │  /api/progress|submissions|notes|      │
                    │  dashboard|guest-progress              │
                    │  /internal/auth/*(認証用DB操作)        │
                    └─────────────────────────────────────┘
```

### 設計の要点

| 論点 | 決定 |
|---|---|
| ルーティング | worker-appが全リクエストを受け、`/api/auth/*` 以外の `/api/*` を service binding 経由で worker-api へフォワード。公開URLは1つのまま(カスタムドメイン不要) |
| セッション | Auth.jsを**JWT戦略**に統一(DBセッション廃止)。worker-apiは同一 `AUTH_SECRET` でCookie内JWTを検証するミドルウェアを持つ。これによりworker-apiは認証状態を自己完結で判定でき、Workerを跨ぐセッション参照が不要になる |
| 認証のDB操作 | Credentials照合・OAuthアカウントupsertは worker-api の `/internal/auth/*` に置く。**service binding経由でのみ到達可能**(worker-apiは公開ルートを持たないため、内部エンドポイントの露出リスクなし) |
| Prismaの所在 | **worker-apiのみ**。worker-appからPrisma依存を完全除去(最大の削減要因) |
| Sentry | サーバ側は `@sentry/nextjs` のサーバSDKを使わず、**Workers向け軽量クライアント(toucan-js)を両Workerに採用**。クライアント側(ブラウザ)は通常のSentry SDKのまま |
| DBコネクション | 変更なし(Neon pooled接続。Worker数が2になってもプールが吸収) |

### 却下した代替案

| 代替案 | 却下理由 |
|---|---|
| Workers Paid移行のみ($5/月) | 資金解決は可能だが、①サイズ予算という構造問題が残る ②Paidの10MiBも保証ではない ③API本数が少ない今が分割の最安時期。なおPaid移行自体は引き続き有効な追加手段(分割後も逼迫したら実施) |
| 3分割(auth独立) | Auth.jsとNext.jsの密結合により、切り出しコスト(コールバックURL・CSRF・ページ側セッション参照の複雑化)が利得を上回る |
| 重量依存の総置換(Prisma→Kysely等) | 効果は大きいが既存コード・テストの改修範囲が広い。分割で足りる間は温存し、worker-apiが単独で予算超過した時の次手とする |

## 3. サイズ予算(正式なアーキテクチャ制約)

| Worker | 予算(gzip後) | CI警告 | CI失敗 | 想定実測 |
|---|---|---|---|---|
| worker-app | 2.5MiB | 2.0MiB | 2.5MiB | Prisma除去後 1.2〜1.5MiB見込み |
| worker-api | 2.5MiB | 2.0MiB | 2.5MiB | Hono+Prisma(small)+zod で 1.0〜1.3MiB見込み |

- CIの `check:bundle-size` をWorker別のマトリクスに改修し、**PRごとに両Workerの実測値と残余裕をコメント出力**する
- 予算改定はADR起票を必須とする(なし崩しの上限緩和を防ぐ)

## 4. タスク分割(T-501〜T-505)

Phase 2開始前に実施する。T-501→T-502→T-503は直列(依存あり)、T-504は503後、T-505は独立(504と並列可)。

### T-501 worker-api骨格 + CI二重化(サイズM / 25 turns)
- 成果物: `workers/api/` にHono骨格(healthエンドポイント、JWT検証ミドルウェア、lib/contracts参照)、wrangler設定2本化(worker-app/worker-api + service binding定義)、CIのbundle-sizeゲートWorker別マトリクス化
- 受入基準: 両Workerが `wrangler deploy --dry-run` 成功、worker-appからbinding経由でhealthが200、既存全テスト緑、CI出力に両Workerのgzipサイズ表示
- Out of Scope: 既存APIの移設

### T-502 API移設(サイズL / 40 turns)
- 成果物: progress/submissions/notes/dashboard/guest-progress の5ハンドラをworker-api(Hono)へ移設、worker-app側 `/api/*` フォワーダ、JWT検証によるユーザー特定
- 受入基準: **既存のAPI統合テストが変更なしで全緑**(外部から見た挙動の完全互換が移設の合格条件)、E2E主要動線が緑
- Out of Scope: 認証のDB操作移設(T-503)

### T-503 認証のJWT化とDB分離(サイズM / 25 turns)
- 成果物: Auth.jsセッションをJWT戦略へ統一、`/internal/auth/verify-credentials` `/internal/auth/oauth-upsert` をworker-apiに実装、worker-appの認証ルートをbinding呼び出しに変更、**worker-appのpackage依存からPrismaを除去**
- 受入基準: 認証の統合テスト(サインアップ/ログイン/OAuth有無/レート制限)が全緑、`grep`でworker-appバンドルにPrismaが含まれないこと、worker-appのgzipサイズが予算内であることをCI出力で確認
- Out of Scope: 既存ユーザーデータの移行(JWT化はセッションのみの変更でありユーザーテーブルは不変)

### T-504 secrets再配置とデプロイ検証(サイズS / 15 turns)
- 成果物: シークレットのWorker別再配置(`DATABASE_URL`→worker-apiのみ / `AUTH_SECRET`→両方 / OAuth系→worker-appのみ)、デプロイワークフローの2 Worker対応、デプロイ後スモークテスト(health+主要動線)
- 受入基準: 本番相当環境で両Workerがデプロイされスモーク緑、worker-apiに公開ルートが存在しないことの確認
- 人間作業: `wrangler secret put` のWorker別再実行(07 A-8の要領。--name を worker 別に指定)

### T-505 Sentry軽量導入(サイズS / 15 turns)
- 成果物: 両Workerにtoucan-js導入(エラー捕捉+リクエストコンテキスト)、クライアント側は通常SDK、02 §10のログ方針との整合
- 受入基準: 意図的エラーがSentryに届くことの確認手順が報告されている、両Workerのサイズ増分が各+100KiB以内

## 5. 撤退基準の更新(07 E-5 / ADR-007 §4.3 を改訂)

| 状況 | 対応 |
|---|---|
| いずれかのWorkerが2.5MiB(gzip)超 | まず依存分析タスク(Bundle Slim同型)→削減不能ならWorkers Paid($5/月) |
| worker-api単独で逼迫が継続 | Prisma→Kysely置換のADR起票(B2) |
| リクエスト数がFree枠(10万/日)接近 | Workers Paid(従来どおり) |
