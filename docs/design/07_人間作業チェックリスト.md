### A. 詳細手順(各サービス公式ドキュメント準拠 / 2026-07 時点)

※各サービスのUIは変わり得るため、手順が合わない場合は併記の公式Docを正とすること。取得した秘密情報は即座にA-8の保管フローへ。

#### A-1 GitHubリポジトリ作成+保護設定(約15分)

1. https://github.com/new でリポジトリ作成(例: `ddia-learning-lab`)。READMEなしの空で作成し、ローカルの初回コミットをpushする運用でよい
2. Settings → Actions → General で「Allow all actions」を確認(既定で有効)
3. **mainブランチ保護**: Settings → Rules → Rulesets → New ruleset(または Settings → Branches のbranch protection rule)で、対象ブランチ `main` に対し「Require a pull request before merging」「Require status checks to pass before merging」を有効化。ステータスチェック検索欄でCIのジョブ名(例: `ci`)を選択 — **選択肢に出るのは過去7日以内に一度実行されたチェックのみ**なので、この項目だけは T-002(CI)完走後に戻ってきて設定する
4. あわせて「Require branches to be up to date before merging」を有効化(古いブランチのままのマージを防止)

- ⚠️ **プラン注意**: ブランチ保護/Rulesetsが使えるのは「公開リポジトリ(Free)」または「私有リポジトリ+Pro以上」。**Free+privateで始める場合は保護ルールが適用できない**ため、公開にするか、Proにするか、当面は運用規律(04のフロー)でカバーするかをここで決める
- 公式Doc: https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-protected-branches/managing-a-branch-protection-rule

#### A-2 Cloudflareアカウント+Wrangler認証(約10分)

1. https://dash.cloudflare.com/sign-up でアカウント作成(メール認証まで完了させる)
2. ダッシュボード左メニューの「Workers & Pages」を一度開き、Workersを有効化(`<name>.workers.dev` サブドメインを設定)
3. ローカルで `npx wrangler login` を実行 → ブラウザが開き許可を求められるので承認(このOAuth認証は人間が行う。CI用は後述A-8のAPIトークン方式)
4. `npx wrangler whoami` でアカウントIDが表示されることを確認(このAccount IDは後でデプロイ設定に使うので控える)

- 公式Doc: https://developers.cloudflare.com/workers/get-started/guide/

#### A-3 Neonプロジェクト作成+接続文字列(約10分)

1. https://neon.com でSign up(メール/GitHub/Google)
2. Console → Projects → **New Project**: プロジェクト名、Postgresバージョン、リージョン(**アプリ利用者に近いリージョン**。日本向けならAWS ap-southeast-1等の近接リージョン)を選び Create Project。`neondb` という初期データベースが作られる
3. Project Dashboard の **Connect** ボタン → モーダルでブランチ/データベース/ロールを選ぶと接続文字列が表示される。形式: `postgresql://<user>:<password>@<endpoint>.neon.tech/<db>?sslmode=require&channel_binding=require`
4. **pooled / direct の2種を両方控える**: 既定表示はホスト名に `-pooler` が付くプール接続(アプリ実行用)。Prismaのマイグレーション実行にはdirect接続を使うのが安全なため、モーダルで切り替えて両方取得 → `DATABASE_URL`(pooled)と `DIRECT_URL`(direct)としてA-8へ
5. 無料プランの制限値(ストレージ/コンピュート時間)をPlansページで確認し、G-1(週次監視)の基準としてメモ

- 公式Doc: https://neon.com/docs/get-started/connect-neon

#### A-4 R2バケット作成+アクセスキー発行(約10分)

1. Cloudflareダッシュボード → **R2 Object Storage**(初回はR2の有効化手続きあり。無料枠内でも支払い方法の登録を求められる場合がある)
2. **Create bucket** → バケット名(例: `ddia-certificates`)、ロケーション、ストレージクラスを選択して作成(CLI派は `npx wrangler r2 bucket create ddia-certificates` でも可)
3. アクセスキー発行: R2 Overview → **Manage API tokens** → **Create Account API token** → 権限は「Object Read & Write」を選び、**対象バケットを作成したバケットのみにスコープ**する → 作成
4. 表示された **Access Key ID / Secret Access Key** を控える。⚠️ **Secret Access Keyはこの画面を閉じると二度と表示されない** → 即A-8へ

- 補足: Workerからのアクセスは基本的にwrangler設定のR2バインディング(キー不要)で行い、このキーはGitHub Actions(修了証ジョブ)等の外部からのS3互換アクセス用
- 公式Doc: https://developers.cloudflare.com/r2/api/tokens/

#### A-5 GitHub OAuth App作成(約10分)

1. GitHub右上アイコン → Settings → 最下部 **Developer settings** → **OAuth Apps** → **New OAuth App**
2. 開発用として登録: Application name `DDIA Lab (dev)` / Homepage URL `http://localhost:3000` / **Authorization callback URL** `http://localhost:3000/api/auth/callback/github`(Auth.jsの規約パス)
3. 作成後の画面で **Generate a new client secret** → Client IDとSecretを控える(⚠️ Secretは生成直後のみ表示)
4. **本番用に同じ手順でもう1つ作成**(OAuth Appのコールバックは実質1系統のため、dev/prodでApp自体を分けるのが定石): callbackは `https://<本番ドメイン>/api/auth/callback/github`(A-9確定後でよい)

- 公式Doc: https://docs.github.com/en/apps/oauth-apps/building-oauth-apps/creating-an-oauth-app

#### A-6 Google OAuthクライアント作成(約15分)※後回し可

現在のGoogle Cloud Consoleは「**Google Auth Platform**」(Branding / Audience / Data Access / Clients)に再編済み。古い解説記事の「OAuth同意画面」メニューは存在しないので注意。

1. https://console.cloud.google.com で新規プロジェクト作成(課金アカウントの紐付けは不要)。作成後、**プロジェクトセレクタで新プロジェクトに切り替わっていることを必ず確認**
2. メニュー → **Google Auth Platform** → 「未構成」の案内が出たら **Get started**: App name / User support email を入力 → **Audience は External** を選択 → 連絡先メールを入れて作成
3. **Clients** → **Create client** → Application type: **Web application** → Authorized redirect URIs に `http://localhost:3000/api/auth/callback/google` を追加(Googleは1クライアントに複数URI登録可なので、本番URLも後でここに追記すればよい)→ 作成し Client ID / Client Secret を控える
4. Data Access(スコープ)は基本プロフィール+メールのみなら追加設定不要。Audience → Publishing status が「Testing」の間は登録したテストユーザーしかログインできないため、公開時に「In production」へ変更する(F-5でやる)

- 公式Doc: https://developers.google.com/identity/gsi/web/guides/get-google-api-clientid

#### A-7 Sentryセットアップ(約10分)

1. https://sentry.io でサインアップ(無料のDeveloperプラン)→ Organization作成
2. **Create Project** → プラットフォームは **Next.js** を選択 → プロジェクト名を入れて作成
3. 表示されるDSN(`https://xxx@xxx.ingest.sentry.io/xxx`)を控える(後からは Project Settings → Client Keys (DSN) で確認可)
4. SDK組み込み自体はSonnetのタスク(コード側)なので、人間はDSNの取得とA-8への登録まで

- 公式Doc: https://docs.sentry.io/platforms/javascript/guides/nextjs/

#### A-8 秘密情報の3箇所への配置(約20分)

**登録先と用途の対応**:
| 秘密情報 | ローカル`.env` | GitHub Actions Secrets | Cloudflare(wrangler secret) |
|---|---|---|---|
| DATABASE_URL / DIRECT_URL | ✓ | ✓(migration/E2E用) | ✓ |
| AUTH_SECRET(`openssl rand -base64 32`で人間が生成) | ✓ | ✓ | ✓ |
| GITHUB_ID / GITHUB_SECRET | ✓(dev App) | — | ✓(prod App) |
| GOOGLE_ID / GOOGLE_SECRET | ✓ | — | ✓ |
| R2 Access Key ID / Secret | — | ✓(修了証ジョブ用) | —(バインディングで代替) |
| SENTRY_DSN | ✓ | ✓ | ✓ |
| CLOUDFLARE_API_TOKEN(デプロイ用) | — | ✓ | — |

手順:

1. ローカル: リポジトリ直下に `.env` を作成して記入(`.gitignore` に含まれることをB段階で必ず確認)
2. GitHub: リポジトリ → Settings → **Secrets and variables → Actions** → **New repository secret** で上表の該当分を登録
3. Cloudflare: プロジェクトディレクトリで `npx wrangler secret put AUTH_SECRET` のように1件ずつ実行(値はプロンプトに貼り付け。コマンド履歴に残さない)
4. デプロイ用 `CLOUDFLARE_API_TOKEN` は、ダッシュボード右上プロフィール → **My Profile → API Tokens → Create Token** → テンプレート「Edit Cloudflare Workers」で発行してGitHub Secretsへ

- 公式Doc: https://docs.github.com/en/actions/security-for-github-actions/security-guides/using-secrets-in-github-actions / https://developers.cloudflare.com/workers/configuration/secrets/

#### A-9 (任意)独自ドメイン(約15分+DNS浸透待ち)

1. Cloudflareダッシュボード → **Domain Registration → Register Domain** で取得(他社取得済みならネームサーバをCloudflareへ変更)
2. 公開時: Workersプロジェクトの **Settings → Domains & Routes → Add → Custom domain** でドメインを割り当て(DNS/証明書は自動設定)
3. ドメイン確定後、A-5(本番OAuth App)/A-6(redirect URI追記)/F-5 を更新

- 公式Doc: https://developers.cloudflare.com/workers/configuration/routing/custom-domains/

#### A-10 Claude Code環境確認(約15分)

1. **プラン確認**: Claude Codeの利用にはClaudeの有料サブスクリプション(またはAPI課金)が必要。自分のプランで利用可能か https://support.claude.com で確認
2. **インストール**: 公式セットアップページ https://code.claude.com/docs/en/setup の手順に従う。現在は**ネイティブインストーラが推奨**(従来の `npm install -g @anthropic-ai/claude-code` は同じネイティブバイナリを入れるが、レガシー扱い。npm版から移行する場合は `claude install` を実行)
3. `claude doctor` でインストール状態を診断、`claude --version` で **v2.1.139以上**(Agent view要件)を確認。古い場合は `claude update`
4. 任意のディレクトリで `claude` を起動し、ブラウザ経由のログインを完了させる(この認証も人間の作業)
5. 使用量/レートリミットの確認方法をこの時点で把握しておく(06 §7: 並列実行はセッション数分クォータを消費するため)

- 公式Doc: https://code.claude.com/docs/en/setup

## 付録: 人間の総作業量サマリ

| フェーズ             | 人間の実働目安                                                                                 |
| -------------------- | ---------------------------------------------------------------------------------------------- |
| A+B 初期セットアップ | 1〜1.5日(1回のみ)                                                                              |
| C 開発サイクル       | 1タスクあたり15〜25分 × 約60タスク ≒ **15〜25時間**(全Phase合計。並列時もレビューは直列で発生) |
| D フェーズ節目       | 各0.5日 × 5回                                                                                  |
| E 意思決定           | 各30分〜(7件)                                                                                  |
| F 公開前             | 2〜3日(法務・校閲・実測)                                                                       |
| G 運用               | 週1〜2時間                                                                                     |
