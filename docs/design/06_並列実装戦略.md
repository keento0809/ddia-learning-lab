## 1. 公式が提供する並列実行の選択肢(調査結果)

公式ドキュメント「Run agents in parallel」は、並列化の手段として次の4方式+支援機能2つを定義している。

| 方式                          | 概要                                                                                                                                      | 特性                                                                                                                                                                                                                         |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ① Subagents                   | 1セッション内部の委任ワーカー。独自コンテキストで側面タスクを実行し要約を返す                                                             | 調整役はセッション内のClaude。ワーカー同士は会話できない(親に報告のみ)                                                                                                                                                       |
| ② Agent view(`claude agents`) | バックグラウンドセッション群を1画面でディスパッチ・監視。**Research preview**                                                             | 調整役は人間。各セッションは独立したフルセッションで、**ファイル編集前に自動で専用worktree(`.claude/worktrees/`)へ隔離**される。ターミナルを閉じても継続                                                                     |
| ③ Agent teams                 | リード+チームメイトが共有タスクリストとメッセージングで協調。**Experimental・既定で無効**(`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`が必要) | 調整役はClaude(リード)。公式が明記する既知の制限: タスク完了ステータスの反映遅れ、セッション再開でチームメイトが復元されない、リードが完了前に切り上げる等。**チームメイトはworktree隔離されない**(ファイル分担は自前で担保) |
| ④ Dynamic workflows           | スクリプトが多数のサブエージェントを実行し、結果を相互検証。数百ファイル規模の移行・全体監査向け                                          | 調整役はスクリプト。1ターンずつの判断では大きすぎる仕事用                                                                                                                                                                    |
| 支援: Worktrees               | セッションごとに独立したgitチェックアウトを与え、同一ファイルの衝突を防止                                                                 | Agent viewは自動適用。削除時にworktreeも消えるため**push/mergeしてから削除**                                                                                                                                                 |
| 支援: `/batch`                | 1つの大きな変更を**5〜30個のworktree隔離サブエージェント**に分割し、各自がPRを開くパッケージ済みスキル                                    | 定型・同型の量産作業向け                                                                                                                                                                                                     |

コスト注意(公式): 複数セッション/サブエージェントの同時実行は**トークン消費を並列数分倍増**させる。

## 2. 本プロジェクトへの適合判断(選定)

### 2.1 選定の判断基準

本プロジェクトは 04文書で「1タスク=1PR」「機械検証可能な受入基準」「人間レビューゲート」「スコープ逸脱の検知」を統制の柱としている。したがって並列化方式は、(a)ファイル衝突を構造的に防げること、(b)タスク単位の検証・レビューを壊さないこと、(c)安定版機能であること、を満たす必要がある。

### 2.2 決定: **ハイブリッド構成**

| 用途                                           | 採用方式                                      | 理由                                                                                                                                                                                                 |
| ---------------------------------------------- | --------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **機能実装タスクの並列化(主軸)**               | **② Agent view + worktree自動隔離**           | 04の「1タスク=1セッション」をそのまま並列に多重化できる唯一の方式。worktree自動隔離でファイル衝突を構造的に排除し、ディスパッチ→監視→介入→レビューの流れが本プロジェクトの人間ゲート運用と完全に一致 |
| **各タスクセッション内部の品質向上**           | **① カスタムSubagents**(読取専用レビュアー等) | 設計書照合・テスト改変検査・大量ログ調査を側面タスクとして分離し、メイン文脈の汚染を防ぐ。ツール制限(読取専用)で安全性を担保                                                                         |
| **教材24本の量産(T-110/111/210s/301)**         | **`/batch`**(または④ Dynamic workflows)       | 同型テンプレートの大量生産という`/batch`の設計意図に合致。5〜30のworktree隔離サブエージェントが各自PRを作るため、モジュール単位レビューも維持される                                                  |
| **Phase 4 の横断監査(a11y/i18n整合/用語統一)** | **④ Dynamic workflows**                       | 全コンテンツ×複数観点の網羅チェック+相互検証という「1ターンずつでは大きすぎる」仕事に該当                                                                                                            |
| **③ Agent teams**                              | **主軸としては不採用**(任意の限定実験のみ)    | Experimentalで既定無効な上、公式明記の制限(タスク状態の遅延・再開不可・worktree非隔離)が本プロジェクトの統制要件(検証可能性・確実な完了判定)と直接衝突するため。安定化後に再評価(§7)                 |

## 3. 並列実行計画: WBSのDAGを「Wave」に変換

**並列の単位はWBSのタスクのまま**。DAG上で依存のないタスクを同一Waveとして同時ディスパッチする。**推奨同時実行数は3〜5**(トークン消費の倍増と、人間レビューが律速になるため。それ以上はレビュー待ち行列が伸びるだけで速くならない)。

```
Phase 0(基盤)
  Wave 0 [直列]   : T-000 → T-001 → T-002 → T-010   ※土台とコントラクトは並列化しない
  Wave 1 [並列3]  : T-003(i18n) ∥ T-004(DB) ∥ T-006(コンテンツPL)
  Wave 2 [並列2]  : T-005(認証) ∥ T-007(レイアウト)

Phase 1(MVP)
  Wave 3 [並列3レーン]:
    レーンA: T-101→T-102→T-103   (画面系: 1セッションが順に消化 or タスク毎に再ディスパッチ)
    レーンB: T-104→T-109          (API系)
    レーンC: T-107a→T-107b→T-107c (実行エンジン)
  Wave 4 [並列3]  : T-105 ∥ T-106 ∥ T-108   (レーン合流後)
  Wave 5 [/batch] : T-110-1〜4(Part I 教材JA 4本を一括量産)→ レビュー後 T-111(EN)を同様に
  Wave 6 [並列2]  : T-112 ∥ T-113

Phase 2
  Wave 7 [並列2]  : T-201→202 ∥ T-203
  Wave 8 [並列5]  : T-204 ∥ T-205 ∥ T-206 ∥ T-207 ∥ T-208   ※Viz5本は完全独立=並列化の最大効果点
  Wave 9 [/batch] : T-210s 教材10本

Phase 3
  Wave 10 [並列4] : T-302 ∥ T-303 ∥ T-305 ∥ T-307 → T-304, T-306 → 教材(/batch)

Phase 4
  Wave 11 [workflows]: 横断監査(a11y・i18n整合・用語統一)→ 検出結果を修正タスクとして再ディスパッチ
```

**マージ規律**: Wave内の完了順にレビュー→mainへrebaseマージ。次のWaveのディスパッチは、依存する成果物がmainに入ってから行う(worktreeはディスパッチ時点のmainから切られるため、依存物が未マージだと妄想実装の温床になる)。

## 4. セットアップ手順(初回のみ・人間が実施)

```bash
# 1) バージョン確認(Agent viewは v2.1.139+ が必要。research previewのためUIは変わり得る)
claude --version

# 2) リポジトリで一度インタラクティブに起動し、権限を承認しておく
#    (--permission-mode を背景セッションに適用するには、事前に対話で一度承認が必要)
cd ddia-learning-lab && claude

# 3) worktreeへコピーすべき未追跡ファイル(.env等)があれば .worktreeinclude を設定

# 4) カスタムサブエージェント定義を配置(§5)
#    .claude/agents/ 配下。/agents はパネルではなくファイル場所の案内を出す仕様のため、
#    ファイルを直接作成するか、Claudeに作成を依頼する
```

### CLAUDE.md への追記(並列運用ルール)

```markdown
## 並列実行ルール(追加)

8. あなたは背景セッションとして専用worktreeで動作している場合がある。
   自分のタスク(T-xxx)のスコープ外のファイルには一切触れない。
   worktreeの外(他セッションの作業)を推測して先回りしない。
9. 作業完了時は必ずブランチをコミット+pushしてから完了報告する
   (セッション削除でworktreeは消えるため、push漏れ=成果物消失)。
10. 依存する型・API・関数がリポジトリに存在しない場合、それは依存タスクが
    未マージである。実装せず「依存未充足」として停止・報告する。
```

## 5. カスタムSubagents定義(各タスクセッション内の品質装置)

`.claude/agents/` に以下の2つを常備する(Markdown+YAMLフロントマター形式、ツール制限付き)。

### spec-checker.md(設計照合・読取専用)

```markdown
---
name: spec-checker
description: 実装が設計書(docs/design/)と一致しているか照合する。実装前の設計読解や、実装後の差異検出を依頼されたときに使う。
tools: Read, Grep, Glob
---

あなたは設計照合の専門家。docs/design/ 配下の該当セクションと
指定された実装ファイルを突合し、「一致 / 差異(仕様側が正か実装側が正か
の判断材料付き) / 設計に記載なし」の3分類で簡潔に報告する。
コードの修正提案はするが、自分では書き換えない。
```

### test-integrity-reviewer.md(テスト誠実性検査・読取専用)

```markdown
---
name: test-integrity-reviewer
description: テストの改変・弱体化(skip, only, expect緩和, timeout延長)や、モック/TODOによる偽装実装を検出する。タスク完了報告の直前に必ず使う。
tools: Read, Grep, Glob, Bash(git diff:*)
---

git diff を確認し、(1) テストの弱体化 (2) 未実装の偽装 (3) スコープ外の
変更ファイル (4) lib/contracts/ への変更 を検出して列挙する。
問題なしの場合も「検査済み・問題なし」と明示的に報告する。
```

**運用**: 04のT2テンプレート最終手順に「完了報告の前に test-integrity-reviewer サブエージェントで自己検査すること」を追記する。これにより04 §6(人間レビュー観点)の1・3・5が機械化され、人間レビューの負荷が下がる=並列時の律速が緩和される。

## 6. 日常の作業手順(1 Waveの回し方)

### Step 1: ディスパッチ(3通り、いずれも同じ)

```bash
# シェルから直接(タスクごとに実行。--name で行を識別しやすくする)
claude --bg "docs/design/03_実装タスク分割書.md の T-204 を、docs/design/04 の
T2テンプレート手順に従って実装してください。ブランチ: feat/T-204-lsm-viz。
計画承認ゲートは省略し、完了報告前に test-integrity-reviewer で自己検査すること。" \
  --name "T-204 LsmTreeViz"

# または claude agents を開き、入力欄にタスクプロンプトを打って Enter(連続投入可)
# または対話セッション中に /bg で現在の会話ごと背景化
```

- Wave 8 なら上記を T-204〜T-208 の5本連続で投入する。各セッションは**ファイル編集前に自動で自分専用のworktreeに移る**ため、5本が同じリポジトリでも衝突しない
- 04で「計画承認ゲートあり」とした複雑タスクは、ディスパッチ後に計画提示で「入力待ち」状態になるので、Step 2のpeekで承認を返す

### Step 2: 監視と介入

```bash
claude agents   # 1画面に全セッション。「Needs input / Ready for review」が上部に固定表示
```

- 行を選んで **Space = peek**(会話に入らず最後の出力・質問を見て返信)、**Enter = attach**(フル会話に入る)
- `/tasks` はセッション内部のタスク一覧、`claude agents` はセッション横断の一覧(用途が別)
- 注意: セッションはマシンのスリープ/シャットダウンでは生き残らない。復帰後は `claude respawn --all`

### Step 3: 完了処理(1セッションごと)

1. 完了報告(04所定フォーマット)と test-integrity-reviewer の検査結果を確認
2. **人間レビュー**(04 §6の観点表)— 受入コマンドはCI or ローカルで必ず再実行
3. push済みブランチをPR化 → mainへrebaseマージ
4. **マージ後にセッションを削除**(先に削除するとworktreeごと成果物が消える)
5. `docs/tasks/STATUS.md` 更新(次のWaveの前提を明確化)

### Step 4: 教材量産Wave(/batch)

```bash
# mainに執筆規約(docs/content-style-guide.md)と模範モジュールが入った状態で:
claude
> /batch Part I の残り教材モジュール(02-data-models, 03-storage, 04-encoding)を
  docs/design/03 の T-110-2〜4 の定義どおりに作成してください。
  1モジュール=1サブエージェント=1PRとし、各PRは npm run validate:content と
  模範解答のgrader passテストを通すこと。
```

`/batch` はworktree隔離されたサブエージェント群がそれぞれPRを開くため、Step 3のレビュー動線がそのまま使える。

## 7. リスクと運用ガードレール

| リスク                               | 対策                                                                                                                                                                                                                       |
| ------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| トークン消費が並列数分倍増(公式明記) | 同時実行は3〜5本まで。Waveの粒度で制御し、使用量を確認しながら投入                                                                                                                                                         |
| 未マージ依存による妄想実装           | Waveの境界=マージの境界を厳守(§3)。CLAUDE.md規則10で停止を義務化                                                                                                                                                           |
| worktree削除による成果消失           | 「push→レビュー→マージ→削除」の順序を固定(§6 Step 3)。CLAUDE.md規則9                                                                                                                                                       |
| Research preview仕様変更(Agent view) | 手順のコマンドが変わった場合は公式docs(/en/agent-view)を再確認。**代替経路**: Agent viewが使えなくても、`git worktree` を手動で切って複数ターミナルで通常セッションを並列起動すれば同じ運用が成立する(worktreesは独立機能) |
| Agent teamsへの誘惑                  | 現時点で不採用(§2.2)。安定版化・worktree隔離対応・タスク状態の信頼性改善が確認できたら、Wave内レーン(例: 画面+API+エンジンの協調)への適用を再評価するADRを起票                                                             |
| レビュー律速                         | サブエージェントによる自己検査(§5)で人間レビューを差分確認に軽量化。それでも詰まる場合は並列数を下げる(並列数を上げる方向で解決しない)                                                                                     |

## 8. まとめ: 何がどう速くなるか

| 局面            | 従来(04の直列運用)  | 本戦略                                           |
| --------------- | ------------------- | ------------------------------------------------ |
| Phase 0         | 直列9タスク         | ほぼ同じ(土台は直列が正しい)。Wave 1-2のみ並列化 |
| Phase 1 機能    | 直列11タスク        | 3レーン並列+合流で**約1/2〜1/3の壁時計時間**     |
| Viz 5本(Wave 8) | 直列5タスク         | **5並列で約1/5**(完全独立のため並列化効果が最大) |
| 教材24本        | 直列24タスク        | `/batch`でモジュール一括量産(レビューが律速)     |
| 横断監査        | 手動+単発セッション | Dynamic workflowsで網羅+相互検証                 |

統制(1タスク=1PR、受入コマンド、人間ゲート)は04のまま一切緩めず、**「同時に走らせる本数」だけを増やす**のが本戦略の要点である。
